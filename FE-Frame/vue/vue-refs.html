# 组件之间的通讯

# $root

在每一个 new Vue 的子组件中，都可以通过 $root 来访问根实例

# $parent

$parent ，$parent 可以从一个子组件中访问父组件实例。

但并不是所有情况下都可以用 $parent 。

比如父组件为 <map>，子组件为 marker （标记的意思）。marker 可以通过 this.$parent.getMap() 从 <map> 中获取数据。

当 marker 的父组件真正绝对只是 map 的时候，可以使用 $parent ，但如果 marker 可能充当了其他组件的子组件时，此时的 this.$parent.getMap() 获取到的数据却不是 map 中的数据。就像 vue 官方文档中所描述，

<map>
  <region>
    <marker>
    </marker>
  </region>
</map>

marker 想访问 map 的数据变成了 this.$parent.$parent.getMap() 。

在绝大多数情况下，触达父级组件会使得你的应用更难调试和理解，尤其是当你变更了父级组件的数据的时候。当我们稍后回看那个组件的时候，很难找出那个变更是从哪里发起的。

# $ref

使用子组件的时候，为子组件绑定 ref 属性，通过 this.$refs='***' 来访问子组件

当 ref 和 v-for 一起使用时，ref 得到的是包含了对应数据源的这些子组件的数组。如下：

<template>
  <div class="home">
    <HelloWorld v-for="item in aa" :key="item" ref="item">
    </HelloWorld>
  </div>
</template>
<script>
import HelloWorld from '@/components/HelloWorld.vue'
export default {
  name: 'Home',
  data(){
    return{
      aa: ['aa','bb','cc']
    }
  },
  components: {
    HelloWorld
  },
  mounted(){
    console.log(this.$refs)
  }
}
</script>

控制台打印出来的 ref 是三个 HelloWorld 组成的数组

$refs 只会在组件渲染完成之后生效，并且它们不是响应式的。这仅作为一个用于直接操作子组件的“逃生舱”——你应该避免在模板或计算属性中访问 $refs。

# $children

当前实例的直接子组件。

节制地使用 $parent 和 $children - 它们的主要目的是作为访问组件的应急方法。更推荐用 props 和 events 实现父子组件通信

# 依赖注入

拿上面 $parent 例子来说，$parent 无法很好的扩展到更深层级的嵌套组件中，那么就引申出来了依赖注入。

依赖注入用到两个实例选项： provide 和 inject

provide 让我们指定想要提供给后代组件的数据/方法

<map>
  <region>
    <marker>
    </marker>
  </region>
</map>

在map中，新建实例选项provide

provide: function () {
  return {
    getMap: this.getMap
  }
}

在其任何后代组件中，通过inject 来接收指定的想要添加的provide

inject: ['getMap']

实际上，可以把依赖注入看作是一个大范围的 prop ，其特点是，

  • 祖先组件不需要知道哪些后代组件使用它提供的 property，

  • 后代组件不需要知道被注入的 property 来自哪里

依赖注入的负面影响：

然而，依赖注入还是有负面影响的。它将你应用程序中的组件与它们当前的组织方式耦合起来，使重构变得更加困难。同时所提供的 property 是非响应式的。这是出于设计的考虑，因为使用它们来创建一个中心化规模化的数据跟使用 $root (opens new window)做这件事都是不够好的。如果你想要共享的这个 property 是你的应用特有的，而不是通用化的，或者如果你想在祖先组件中更新所提供的数据，那么这意味着你可能需要换用一个像 Vuex (opens new window) 这样真正的状态管理方案了。
